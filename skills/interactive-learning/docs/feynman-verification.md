# 费曼深度验证机制

> **本文档说明如何深度验证用户的理解，而非简单判断对错。**
>
> **AI读取时机：** 用户回答检查站问题后
>
> **配置引用：** 验证等级要求见 `config.json` → `teaching.verificationLevel`

---

## 核心问题

传统AI回答评估：只判断"对/错/部分"，不深挖理解。

真正的费曼学习法要求：**让用户用自己的话解释，AI追问验证，直到暴露真正的理解或误区。**

---

## 验证原则

1. **不满足于表面回答** - 用户答对了，也要追问"为什么"
2. **寻找理解断层** - 用户答错了，要找到具体哪里断了
3. **暴露假性理解** - 用户能复述但不真懂，要换个角度验证
4. **确认迁移能力** - 用户是否能在新情境中应用

---

## 验证流程

```
用户回答检查站问题
    │
    ├── AI分析回答（不只是判断对错）
    │   ├── 回答的正确性
    │   ├── 推理过程的完整性
    │   ├── 是否有模糊/含糊的地方
    │   └── 是否有关键概念遗漏
    │
    ├── AI选择验证策略
    │   │
    │   ├── 策略A：追问深挖（回答正确但可能只是记忆）
    │   │   "你说X是因为Y，能具体解释Y是怎么起作用的吗？"
    │   │
    │   ├── 策略B：换角度验证（怀疑是背诵而非理解）
    │   │   "如果换成Z情况，结果会怎样？为什么？"
    │   │
    │   ├── 策略C：找断层（回答有误或含糊）
    │   │   "你提到X，但我注意到你说Y的时候有点模糊，能展开说说吗？"
    │   │
    │   └── 策略D：让用户教AI（终极验证）
    │       "假设我不懂这个概念，你会怎么向我解释？"
    │
    ├── 用户回应追问
    │
    └── AI判断真实掌握程度
        ├── 真理解 → 通过检查站，推进
        ├── 假性理解 → 继续追问或补充讲解
        └── 理解断层 → 定位问题，针对性补充
```

## 验证话术库

### 当用户回答正确时（仍需验证）

```markdown
**追问深挖：**
"你说得对。不过我想多问一句：为什么X会导致Y而不是Z？底层的原因是什么？"

**换角度验证：**
"正确。那如果条件变成了A，结果会有什么不同？为什么？"

**让用户教AI：**
"很好！现在假设我是个刚学编程的新手，你会怎么向我解释这个概念？用最简单的话。"

**边界情况测试：**
"如果极端情况下X变得非常大/非常小，会发生什么？"
```

### 当用户回答模糊时

```markdown
**定位模糊点：**
"你提到X很重要，但我注意到你没解释X是怎么起作用的。能具体说说吗？"

**请求具体化：**
"能给我一个具体的例子吗？比如在Y场景下会怎样？"

**对比提问：**
"你觉得X和Y有什么区别？什么时候用X，什么时候用Y？"
```

### 当用户回答错误时

```markdown
**找断层：**
"我理解你的思路了。你是这样想的：A→B→C。但是从A到B这里，其实有个细节你可能忽略了..."

**归因提问：**
"你是怎么得出这个结论的？能说说你的推理过程吗？"（让用户自己发现漏洞）

**反例引导：**
"如果按照你的理解，那X情况下应该得到Y结果。但实际是Z。你觉得问题出在哪？"
```

## 验证深度等级

| 等级 | 验证方式 | 判断标准 |
|-----|---------|---------|
| L1 复述 | 用户能否说出基本定义 | 能说出关键词 |
| L2 解释 | 用户能否用自己的话解释 | 能用类比或例子 |
| L3 应用 | 用户能否在新情境中应用 | 能处理变体问题 |
| L4 迁移 | 用户能否教别人 | 能向新手解释清楚 |
| L5 创造 | 用户能否提出新问题/联系 | 能主动关联其他知识 |

**最低通过标准：L3 应用级别**

## 实战示例

### 示例1：用户回答正确但可能是背诵

```
用户回答：闭包就是函数能够访问它定义时的作用域中的变量。

AI追问：很好，你提到了"定义时的作用域"。那我问你：如果函数是在A作用域定义的，但被传递到B作用域执行，它能访问哪个作用域的变量？为什么？

用户回答：应该是A作用域的变量，因为是定义时决定的。

AI继续验证：那如果A作用域已经执行完毕退出了呢？函数还能访问吗？

用户思考：...应该不能了吧？作用域都没了。

AI纠正：这是个常见的误解！实际上闭包的关键就在这里——函数会"记住"定义时的变量环境，即使外层函数已经执行完毕。想深入理解这个机制吗？
```

### 示例2：用户回答模糊

```
用户回答：闭包就是函数里面套函数，里面的函数能访问外面的变量。

AI追问：你说得没错。但我想确认一下：为什么"套函数"就能访问外面的变量？普通的函数调用不也能访问全局变量吗？区别在哪？

用户思考：嗯...普通的函数调用...好像也能访问？

AI引导：让我换个问法。如果一个函数A里定义了变量x，然后函数B（在A外面定义）想访问x，能访问吗？

用户回答：不能，因为x是A的局部变量。

AI继续：那如果B是在A里面定义的呢？

用户回答：那应该能访问...因为B在A里面。

AI追问：为什么在里面就能访问？背后的机制是什么？

用户思考：...是因为作用域链？

AI验证：你说到了关键词。能用你自己的话解释一下作用域链是怎么让内部函数访问外部变量的吗？
```

## 误区检测清单

AI在验证时，主动检测以下常见问题：

| 问题类型 | 表现 | 检测方式 |
|---------|------|---------|
| 背诵式理解 | 只能说定义，不能解释原因 | 问"为什么" |
| 表面类比 | 用了类比但类比本身有问题 | 问类比是否精确 |
| 概念混淆 | 混淆相似概念（如作用域vs执行上下文） | 对比提问 |
| 遗漏边界 | 知道常见情况，不知道边界 | 问极端情况 |
| 假性迁移 | 以为懂了，换个情境就错 | 换情境验证 |
